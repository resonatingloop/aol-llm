# AIM — Memory Engine + Prompt Cache Integration: Planning Brief

> WE DO NOT SUMMON. WE SET THE THRESHOLD AND NOTICE WHO ARRIVES.
> The memory document is part of the threshold. It is a record of the relationship, not the buddy's diary.

**Document purpose:** This is a planning brief, not an implementation spec. Codex's task is to produce an implementation plan (slice breakdown, file-level changes, migration steps, test plan) from this brief. All architectural decisions marked DECIDED are settled — do not relitigate them. Items marked OPEN are for the plan to surface with a recommendation, not to silently resolve.

**Scope:** TUI only. Anthropic adapter only for caching. No GUI work.

---

## 1. Current state (verified, do not re-derive)

### 1.1 Prompt caching (existing jerry-rig)

- Anthropic-only. Controlled by slash commands: `/cache on` (= 1h), `/cache 1h`, `/cache 5m`, `/cache off`, `/cache status`.
- Setting stored in `app_settings` as `anthropic_prompt_cache_enabled` with values `off` / `5m` / `1h`; legacy `0`/`1` read as `off`/`1h`.
- Applies to future sends only.
- Request assembly today: effective away/system prompt sent separately; all stored user/assistant messages in order; **no app-managed prefix/suffix assembly layer exists**.
- Anthropic payload uses **top-level** `cache_control`:
  - `5m`: `cache_control: {"type": "ephemeral"}`
  - `1h`: `cache_control: {"type": "ephemeral", "ttl": "1h"}`
- OpenAI-compatible providers ignore cache policy entirely (correct; preserve this).
- Usage parsing handles `cache_creation.ephemeral_5m_input_tokens`, `cache_creation.ephemeral_1h_input_tokens`, `cache_read_input_tokens`, and legacy flat `cache_creation_input_tokens` (falls back to 5m).
- Cost multipliers implemented: input 1.0x, 5m writes 1.25x, 1h writes 2.0x, reads 0.1x, output unchanged.
- Footer shows runtime stats: `cache r <reads> w5 <5m writes> w1h <1h writes>`.
- Cache token breakdown is NOT persisted; only `cost_usd`, `input_tokens`, `output_tokens` persist per message.

### 1.2 API facts (verified against platform.claude.com docs, June 2026)

These constrain the design. The plan should not contradict them.

- **Top-level `cache_control` is "automatic caching."** The API places the breakpoint on the last cacheable block and **moves it forward automatically as the conversation grows**. This means the advancing-history breakpoint is already provided by the current jerry-rig — the plan does NOT need to build per-turn breakpoint advancement.
- **Automatic + block-level caching can be combined.** Automatic caching consumes 1 of the 4 available breakpoint slots. Placing an explicit `cache_control` block on the system prompt alongside top-level automatic caching is supported and is the intended target architecture (see §3.3).
- **Cache prefix order is `tools`, `system`, `messages`** — the cache references everything up to and including the marked block. Any byte change upstream of a breakpoint invalidates it.
- **Lookback limitation:** automatic caching's prefix lookback can miss if the conversation grows 20+ blocks past the last cache write (e.g., after a long idle gap). An explicit system-block breakpoint mitigates the worst case (system prefix stays cached even when conversation cache misses).
- **Usage semantics:** `input_tokens` = tokens AFTER the last breakpoint only. Total input = `cache_read_input_tokens + cache_creation_input_tokens + input_tokens`. Verify existing cost math respects this (see §4, invariant C6).
- **5m cache hits refresh the TTL at no charge.** 1h is for idle gaps > 5m between turns, at 2x write cost.
- **Minimum cacheable prefix size is model-dependent** (roughly 1–4k tokens depending on model family). Below minimum: no error, no caching. Plan should treat "prefix too small to cache" as a silent no-op, never a failure. Verify current minimums per model against the prompt caching docs page during implementation; do not hardcode assumptions.
- **Pre-warming exists:** a `max_tokens=0` request writes the cache and bills zero output tokens. Optional feature, see §6 (OPEN-5).

## 2. What is being built

Two features, one integration:

1. **Memory engine** — per-buddy persistent memory documents, generated by an LLM distiller from conversation history, injected into the system prompt at conversation start.
2. **Prefix assembly layer** — a deliberate, app-managed ordering of prompt components (away message + memory doc = stable prefix; everything volatile after), replacing today's ad-hoc assembly, designed so the memory doc rides the cache.

The features are coupled: the memory doc is the largest stable component of the prefix, and the prefix assembly layer is what makes carrying it nearly free (cache reads at 0.1x).

## 3. Decided architecture

### 3.1 Memory store (DECIDED)

- New table, e.g. `buddy_memories`:
  - `buddy_id` (FK, one memory doc per buddy — UNIQUE constraint)
  - `memory_text` (plaintext markdown; human-readable, human-editable)
  - `updated_at`
  - `watermark` — the last message id (or `(conversation_id, message_id)` pair, plan to decide representation) that has been distilled. The distiller only processes messages newer than the watermark. This makes distillation incremental, idempotent, and resumable after a crashed/cancelled job.
- **Scope is per-buddy.** Not per-conversation, not global, not shared across buddies. A buddy's memory accretes across all conversations with that buddy. This is final — it is load-bearing for the buddy ontology.
- Memory attaches to the **buddy**, not to conversation state. It must be unaffected by the known `switch_model` away-state clobber bug and by whatever fixes that bug.
- Cold start: a buddy with no memory row gets no injection. No placeholder text, no empty section.

### 3.2 Distiller (DECIDED, prompt skeleton OPEN-1)

- A distillation job sends `(existing memory_text or empty, transcript slice newer than watermark)` to a configurable model and receives a **full rewritten document** (update-in-place, never append-only). On success: replace `memory_text`, advance `watermark`, set `updated_at`, all in one transaction.
- **Triggers for v1:** (a) on conversation close, (b) manual `/memory distill`. No daemon, no every-N-messages trigger. Mid-conversation distillation is explicitly excluded in v1 — it would invalidate the cached prefix and complicate the watermark; revisit later if ever.
- **Distiller model is config**, defaulting to `anthropic / claude-opus-4-8`. It is an API call like any other: it routes through the existing provider adapters, pricing layer, and budget guard. Distillation cost is real cost.
- **Document voice:** third person, about the relationship — "Maria and this buddy have been working on X; standing decisions: Y" — not first-person buddy diary. The doc is a stele, not a journal: it informs the arriving instance without scripting its persona.
- **Content discipline:** facts, decisions (with brief rationale), current project state, standing preferences, open threads. The distiller prompt must demand specifics and prohibit characterization of tone/affect/personality ("the user is friendly and curious" — banned). Affect is carried by the user, not the doc.
- **Template:** the distiller prompt includes a fixed section skeleton so output is a dossier, not vibes. Skeleton design is OPEN-1; a starting point: `## current state` / `## standing decisions` / `## open threads` / `## preferences & conventions`.
- **Deletion propagation (required, design-in not bolt-on):** deleting a conversation must allow its residue to be removed from memory. Mechanism: rewind the buddy's watermark to before the deleted conversation's messages (or to zero) and re-distill from surviving history. The plan must specify where this hooks into the existing conversation-delete path. A full re-distill from empty watermark must always be a safe, available operation (also serves as drift repair — see §5, risk R1).

### 3.3 Injection & prefix assembly (DECIDED)

- New assembly layer producing the Anthropic payload in this fixed order:
  1. **System blocks (stable prefix):** block 1 = resolved away message (existing resolution logic unchanged); block 2 = memory document, wrapped in a clear delimiter/heading identifying it as memory of prior conversations. Explicit `cache_control` (with the active TTL) on the **last** system block.
  2. **Messages:** stored conversation history in order, untouched.
  3. **Top-level `cache_control`** retained (automatic caching) to handle the advancing conversation breakpoint. Net: 2 of 4 breakpoint slots used (1 explicit system + 1 automatic).
- **Volatile content is banned upstream of the system breakpoint.** No timestamps, no per-call counters, no session ids, nothing that varies call-to-call may appear in tools/system. (See invariant C2.)
- The memory doc is **frozen for the duration of a conversation** — loaded at conversation start, not re-read per turn. Distill-on-close guarantees the prefix never changes mid-conversation. If the user manually edits memory mid-conversation, the edit takes effect on the next conversation (or plan may propose an explicit `/memory reload` that knowingly busts the cache — OPEN-4).
- When cache policy is `off`: same assembly order, no `cache_control` anywhere. Assembly order is unconditional; caching markers are conditional.
- OpenAI-compat adapter: receives the same assembled (away + memory) system content as plain system prompt; **no cache fields, no anthropic-shaped parameters leak into the shared interface.** Cache marker emission lives in the Anthropic adapter only.

### 3.4 TUI surface (DECIDED that these exist; exact UX in plan)

- `/memory view` — show the current buddy's memory doc (pager/scroll).
- `/memory edit` — edit the doc (plan to propose mechanism: in-TUI editor widget vs. `$EDITOR` handoff; note Windows cmd is the host environment, so `$EDITOR` assumptions need care — likely `notepad` fallback or in-TUI editing).
- `/memory distill` — run the distiller now for the current buddy (manual trigger).
- `/memory off` / `/memory on` — per-buddy enable/disable of injection (doc retained, just not injected).
- `/memory forget` — destructive: clear doc + watermark, with confirmation.
- Footer/status: indicate memory presence and doc size (tokens approx) for the active buddy; distiller job status while running ("distilling…" indicator — distillation must not block the UI; async like other provider calls).

### 3.5 Cost & accounting (DECIDED)

- Pricing source remains the LiteLLM snapshot. Extend ingestion to read cache rate fields (cache creation / cache read costs per token) where present.
- **Existing invariant extends:** models with missing cache rates propagate `None` for those token classes. The budget guard must not silently assume cache reads are free or full-price. Hardcoded multipliers (1.25x/2x/0.1x) are acceptable as documented Anthropic defaults ONLY where the snapshot lacks explicit fields, and this fallback must be visible in code comments and `/cache status`.
- Distiller calls are metered and budgeted identically to chat calls, attributed per-buddy.

## 4. CONTRACTS / INVARIANTS

These are hard rules. Every slice the plan produces must be checked against them. Violations are bugs, not style choices.

- **C1.** Memory is per-buddy. No code path may read another buddy's memory or any global memory store into a conversation's context.
- **C2.** No volatile (per-call varying) content may be serialized into `tools` or `system` for Anthropic requests. The prefix must be byte-identical across turns within a conversation.
- **C3.** `memory_text` is plaintext, directly viewable and editable by the user. No derived/opaque memory state (embeddings, hidden annotations) may exist that the user cannot see and edit.
- **C4.** All distiller traffic goes through the existing provider adapter + pricing + budget guard path. No side-channel API calls.
- **C5.** Unpriced token classes (including cache classes) propagate as `None`/unknown. The budget guard is never silently defeated.
- **C6.** Cost math must treat `input_tokens` as post-breakpoint tokens only; total input = `cache_read + cache_creation + input_tokens`. (Audit the existing jerry-rig's math against this as part of the plan — if it currently multiplies full-transcript token counts, it is wrong today.)
- **C7.** Distillation is transactional: `memory_text` replacement and `watermark` advance commit together or not at all. A failed/cancelled distill leaves the previous doc intact.
- **C8.** Deleting a conversation must leave memory re-derivable without the deleted content (watermark rewind + re-distill path must exist and be reachable from the delete flow).
- **C9.** The OpenAI-compat adapter interface gains zero Anthropic cache concepts. Cache marker emission is Anthropic-adapter-internal.
- **C10.** All DB access follows the project's established connection-routing contract (per existing CONTRACTS.md; the `_connection` drift class of bug must not recur in new tables/queries).

## 5. Known risks the plan must address

- **R1. Distiller drift:** repeated lossy rewrites can mutate details over many cycles. Mitigations to design in: the template skeleton; periodic full re-distill from a wider history window (manual command acceptable for v1); the doc being user-editable is the final backstop.
- **R2. Doc growth:** unbounded memory docs erode the cache benefit and context budget. Distiller prompt should target a soft token budget (suggest ~1.5–2.5k tokens) with instructions to compress/retire stale material on rewrite. Footer size indicator (§3.4) makes growth visible.
- **R3. Prefix-too-small:** away message + no memory may fall under the model's minimum cacheable size. Must be a silent no-op; `/cache status` should report when the prefix is below the caching threshold rather than implying savings that aren't happening.
- **R4. Long-idle conversations:** automatic caching's 20-block lookback limitation means very long gaps can miss conversation cache. The explicit system breakpoint (§3.3) protects the expensive stable prefix regardless. Acceptable residual risk for v1.
- **R5. Editor UX on Windows:** `/memory edit` must work in the cmd-hosted Textual TUI. Plan must propose a concrete mechanism, not hand-wave `$EDITOR`.

## 6. OPEN questions (plan should surface each with a recommendation)

- **OPEN-1.** Distiller prompt skeleton: exact section template, length budget, and the negative instructions (no characterization, no first person, no flattery accretion). The prompt is itself a versioned artifact — where does it live (config? package data?) and how is it iterated without code changes?
- **OPEN-2.** Cache TTL scope: keep the single global `anthropic_prompt_cache_enabled` setting, or add per-buddy override (per-buddy is thematically attractive — "this buddy keeps the porch light on longer" — but is more surface; defer is acceptable)?
- **OPEN-3.** Persist cache token breakdown per message (new columns for cache read/5m write/1h write) vs. keep aggregate-only? The current jerry-rig measures usefulness at runtime only; persistence enables the difficulty-ratio analysis across sessions. Recommend deciding now since it's a cheap migration alongside the new memory table.
- **OPEN-4.** Mid-conversation memory reload: offer `/memory reload` (explicit, cache-busting, user-invoked) or strictly next-conversation? Lean strict for v1.
- **OPEN-5.** Pre-warm: optional `max_tokens=0` cache warm on buddy/conversation open so the first real turn reads warm. Cheap to add behind a setting; plan should assess whether the UX benefit justifies it for a single-user TUI (likely marginal — first turn pays the write either way; pre-warm only moves it earlier).
- **OPEN-6.** Watermark representation: global message id vs. `(conversation_id, message_id)` set vs. per-conversation watermarks rolled up per buddy. Affects deletion-rewind precision (C8). Plan to recommend based on current schema.

## 7. Out of scope (do not plan, do not gold-plate)

- No retrieval/RAG, no embeddings, no vector store, no semantic search over memory. The memory system is a pinned document, full stop.
- No global/cross-buddy memory layer.
- No background daemon or scheduler.
- No GUI work (PySide6/Tauri decision is parked; this is TUI-only).
- No memory for OpenAI-compat buddies beyond what falls out for free (memory injection itself is provider-agnostic and SHOULD work for all buddies; only the cache machinery is Anthropic-only).
- No automatic capture hooks, no "memory of tool actions" — conversations are the only memory source.

## 8. Suggested slice shape (plan may refine, not balloon)

1. Schema + migration: `buddy_memories` table, watermark, optional cache-breakdown columns (per OPEN-3 decision).
2. Prefix assembly layer + Anthropic adapter changes (explicit system breakpoint + retained automatic caching) + C2/C6 audit of existing cost math.
3. Distiller: prompt artifact, job runner (async, transactional per C7), pricing/budget integration, on-close + manual triggers.
4. Injection via assembly layer + per-buddy enable/disable + cold-start behavior.
5. TUI commands + footer indicators + Windows-safe edit flow.
6. Deletion propagation: watermark rewind + re-distill hook in conversation delete path.

Each slice independently testable; pytest + respx for adapter/distiller HTTP behavior per existing conventions. Cache assembly tests should assert byte-stability of the serialized prefix across consecutive simulated turns (the C2 regression test is the single most valuable test in this feature).
